# Lesson: The Agent Loop

!!! success "This lesson is complete"
    This is your flagship first lesson — read it top to bottom, then answer the checkpoint.

Welcome to the most important idea in all of agentic AI. If you internalize just this one lesson, everything else becomes variations on a theme. We build the concept, then read the *real* implementation in the Pi repo: `packages/agent/src/agent-loop.ts`.

## 1. The one idea

**An "agent" is just a `while` loop wrapped around an LLM that is allowed to call tools.**

```text
loop:
    ask the model what to do next
    if the model asked to use a tool → run it, give it the result, loop again
    if the model just answered → stop
```

A plain chatbot is *one* model call. An **agent** is this loop — the model can act, see the result, and decide what to do next, repeatedly, until the task is done.

!!! note "C++ analogy"
    You've written a message pump before: `while (GetMessage(&msg)) { Dispatch(msg); }`. Same shape. The "events" are generated by a model reasoning about a goal; "dispatch" means executing a tool and feeding the result back. The loop exits when the model decides it's done, not on `WM_QUIT`.

## 2. The primitive: the tool-call cycle

Modern LLMs support **tool calling**:

1. You send the model a list of tools (name, description, parameter schema).
2. Instead of text, the model can reply with a **structured request**: *call `read_file` with `{path: 'config.json'}`*. **The model does not run anything** — it only emits the request.
3. **Your code** executes the tool and sends the result back as a new message.
4. The model sees the result and continues.

> The single most important mechanical fact in the field: **the model decides, your code acts.** The LLM is a reasoning engine with no hands; the loop is its hands.

## 3. The real loop

The heart of `runLoop` in `agent-loop.ts`, reduced to its control flow:

```typescript
let pendingMessages = (await config.getSteeringMessages?.()) || [];

while (true) {                        // OUTER loop: handles follow-ups
    let hasMoreToolCalls = true;

    while (hasMoreToolCalls || pendingMessages.length > 0) {   // INNER: the turn cycle
        const message = await streamAssistantResponse(currentContext, config, ...);

        if (message.stopReason === "error" || message.stopReason === "aborted") {
            /* emit agent_end, return */
        }

        const toolCalls = message.content.filter((c) => c.type === "toolCall");

        hasMoreToolCalls = false;
        if (toolCalls.length > 0) {
            const batch = await executeToolCalls(currentContext, message, ...);
            hasMoreToolCalls = !batch.terminate;
            // push tool results back into the context
        }
    }

    const followUps = (await config.getFollowUpMessages?.()) || [];
    if (followUps.length > 0) { pendingMessages = followUps; continue; }
    break;
}
```

Trace *"read config.json and tell me the port"*:

| Iteration | Model output | toolCalls | hasMoreToolCalls | What happens |
|---|---|---|---|---|
| 1 | call `read_file({path:'config.json'})` | 1 | `true` | run tool, append result, **loop** |
| 2 | "The port is 8080." (text) | 0 | `false` | inner condition fails → **stop** |

!!! danger "The punchline — burn this in"
    The loop continues **as long as the model keeps requesting tools**. The moment it replies with text and *no* tool calls, `hasMoreToolCalls` stays `false` and the loop stops. **The model ends the loop by simply choosing to stop calling tools** — "done" is the *absence* of a tool call, not a flag it sets.

## 4. Three details that pay off later

- **The LLM boundary** (`convertToLlm`): the agent's rich internal messages are converted to what the model understands (`user`/`assistant`/`toolResult`) right before each call. This seam is where a lot of real engineering lives.
- **Streaming**: responses arrive as `text_delta` / `toolcall_delta` events accumulated into one message — that's the token-by-token effect. The loop proceeds only once the message is complete.
- **Two queues**: *steering* (checked each turn — interrupt mid-task) and *follow-up* (the outer loop — queue work for after the agent would stop). This is how a human stays in the loop.
